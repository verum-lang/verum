# AOT mem-suite cross-tier blockers (snapshot 2026-05-23)

Pre-existing AOT-only defects gating `verum test --aot` runs of the
mem conformance suite.  Each fix below unblocks the entire `--aot`
column for mem; closing them is a separate, multi-day effort each.

## Closed

1. **`winsock_read` 2-vs-3-arg mismatch** — commit `259f39b06` routed
   POSIX-compat wrappers through `winsock_recv`/`winsock_send`
   directly, eliminating the 3-arg vs 2-arg signature collision.

2. **`int_encode` aliased-mount** — closed by parallel agent before
   2026-05-23: `core/net/h3/qpack/session.vr:58` mount switched from
   `{encode as int_encode, decode as int_decode}` to bare `{encode,
   decode}` + qualified-path `core.net.h3.qpack.integer.encode(...)`
   call sites.

3. **`cursor_new` 0-vs-1-arg sibling collision** — commit `1c309fd4e`
   aliased to `btree_cursor_new` in
   `core/database/sqlite/native/l4_vdbe/cursor_btree_bridge.vr`,
   disambiguating from the 0-arg `cursor_new` siblings in
   `vtabs/{dbstat,generate_series,json_each}.vr`.

4. **`ws_accept_key` alias-mount** — commit `1c309fd4e` switched
   `core/net/weft/websocket.vr` to qualified-path
   `core.net.websocket.accept_key(&key)`.

5. **`value_from_json` / `value_to_json` alias-mount** — commit
   `1c309fd4e` switched `core/encoding/jsonb.vr` to qualified-path
   `core.encoding.value.{from,to}_json(...)`.

6. **`sleep_seconds` non-existent mount** (real stdlib bug) — commit
   `1c309fd4e` fixed `core/search/meilisearch/client.vr` mount from
   `async_.sleep_seconds` to `core.async.timer.sleep_secs`.

7. **`filename` AOT closure-capture** — commit `30b4fbb0b` hoisted
   f-string out of `.ok_or_else(|| ...)` closure body in
   `core/security/tuf/client.vr::update_targets`.

## Open

### Active blocker: `translate` 1-vs-3-arg sibling collision

**Surfaced:** 2026-05-23 after B6 closed.

```
wrong number of arguments for translate: expected 1, found 3
(in function handle_translate)
```

Same defect class as B3 (`cursor_new`): bare-name `translate` resolves
to a 1-arg sibling instead of the 3-arg target the caller wants.
Multiple `translate` definitions across stdlib.

**Workaround:** rename either the caller's mount or the call site
to a unique qualified path / alias.

### Cascade observation

After closing 7 AOT blockers in this session, the cascade pattern is:

* **Class A — alias-mount fail to register** (B1, B3, B4, B5): the
  `register_import_aliases` walker in
  `crates/verum_vbc/src/codegen/mod.rs:7184` lookups the target
  function in `ctx.functions` AT alias-registration time.  If the
  target isn't yet loaded (topological-order skew), the alias is
  silently dropped, and the call site falls through to undefined.
  **Real fix:** deferred-alias queue processed AFTER all stdlib
  modules' functions are loaded.

* **Class B — bare-name signature collision** (B2 `cursor_new`,
  current `translate`): multiple modules export a function with the
  same simple name but different signatures.  `lookup_function(name)`
  picks first-suffix-wins, often the wrong arity.
  **Real fix:** mount-scope-aware `lookup_function` (task #17/#39).

* **Class C — closure-capture under AOT** (B6 `filename`): closure
  body inside `.ok_or_else(|| ...)` doesn't see enclosing `let`
  bindings.  Workaround: hoist captured values to separate `let`
  bindings before the closure.
  **Real fix:** closure-capture lowering in
  `crates/verum_codegen/src/llvm/` — root in closure-env synthesis.

Each Class C fix is surgical (per-site).  Class A and B require
multi-day architectural refactors of the function-resolution path.

### B1 — `undefined function: int_encode (in function encode_encoder)`

**Surfaced:** 2026-05-23, immediately after B0 (winsock_read) closed.

**Site:** `core/net/h3/qpack/session.vr:58`
```verum
mount core.net.h3.qpack.integer.{encode as int_encode, decode as int_decode};
```

`encode_encoder` (also in session.vr) calls `int_encode(...)` — the
alias to `core.net.h3.qpack.integer.encode`.  Under `--interp` this
resolves via the lookup chain; under `--aot` the vbc_lowering pass
loses the alias mapping and reports "undefined function: int_encode".

**Defect class:** aliased `mount X.{Y as Z};` imports don't propagate
into AOT precompile's function-id remap table.  Same root affects
every `as` alias across stdlib (likely a wide blast radius — grep
for `mount.*\.{.*as ` returns many sites).

**Fix path candidates:**
1. Resolve aliases at vbc_lowering time — extend the user-body
   function-id remap to walk import aliases.
2. Pre-resolve aliases in the AST → flatten `int_encode(args)`
   to `core.net.h3.qpack.integer.encode(args)` before vbc_lowering.

Estimate: 1-2 days of AOT codegen work plus regression-sweep.

## Why the mem suite cares

Mem tests don't directly call `int_encode` or any QPACK code — but
the AOT pipeline compiles ALL stdlib modules in one pass (precompile),
so a defect ANY stdlib module gates ALL `--aot` runs across the
whole codebase.  Mem tests inherit the blocker.
