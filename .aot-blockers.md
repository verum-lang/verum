Pre-existing AOT-only defects gating `verum test --aot` runs of the
mem conformance suite.  Each fix below unblocks the entire `--aot`
column for mem; closing them is a separate, multi-day effort each.

## Open

### B1 — `undefined function: int_encode (in function encode_encoder)`

**Surfaced:** 2026-05-23, immediately after B0 (winsock_read) closed.

**Site:** `core/net/h3/qpack/session.vr:58`
```verum
mount core.net.h3.qpack.integer.{encode as int_encode, decode as int_decode};
```

`encode_encoder` (also in session.vr) calls `int_encode(...)` — the
alias to `core.net.h3.qpack.integer.encode`.  Under `--interp` this
resolves via the lookup chain; under `--aot` the vbc_lowering pass
loses the alias mapping and reports "undefined function: int_encode".

**Defect class:** aliased `mount X.{Y as Z};` imports don't propagate
into AOT precompile's function-id remap table.  Same root affects
every `as` alias across stdlib (likely a wide blast radius — grep
for `mount.*\.{.*as ` returns many sites).

**Fix path candidates:**
1. Resolve aliases at vbc_lowering time — extend the user-body
   function-id remap to walk import aliases.
2. Pre-resolve aliases in the AST → flatten `int_encode(args)`
   to `core.net.h3.qpack.integer.encode(args)` before vbc_lowering.

Estimate: 1-2 days of AOT codegen work plus regression-sweep.

### Subsequent

Closing B1 will likely surface the next cascade defect.  Re-run

## Why the mem suite cares

Mem tests don't directly call `int_encode` or any QPACK code — but
the AOT pipeline compiles ALL stdlib modules in one pass (precompile),
so a defect ANY stdlib module gates ALL `--aot` runs across the
whole codebase.  Mem tests inherit the blocker.
